ТЗ: Полная конфиденциальность данных и защита приложения от просмотра в браузере
🎯 Цель

Обеспечить, чтобы при открытии приложения через браузер (не через Telegram) и при любом взаимодействии:

не были доступны, видны или обнаружимы:

данные администратора,

секретные ключи,

токены бота,

переменные окружения,

внутренние URL,

API ключи,

структура backend,

любая конфигурация приложения,

endpoints,

бизнес-логика,

идентификаторы систем,

имена таблиц/коллекций,

учетные данные,

любые другие приватные данные.

📍 Что именно должно быть реализовано
1️⃣ Абсолютная защита от просмотра кода/данных через браузер
Frontend

Клиентская часть (Mini App UI) не хранит и не отдает:

BOT_TOKEN

CRYPTOBOT/CRYPTOPAY токен

MONGO_URI

JWT_SECRET

любые другие секреты

Никакая переменная окружения не включается в финальный сборочный файл (bundle).

Никакой endpoint, ключ или строка не попадает в клиентский JavaScript.

Все секреты используются только на сервере, и ни один из них не виден в network calls (FE → BE), браузерных логах или DevTools.

2️⃣ При открытии через браузер без Telegram

Если приложение открыли не через Telegram, то:

Нужно показывать только статичный экран:

Откройте приложение через Telegram


Без каких-либо:

сетевых запросов,

графиков,

логики,

API вызовов,

доступа к backend,

каких-либо данных,

UI элементов, кроме одной кнопки:
«Открыть в Telegram»

И обязательно:

выводить только статический код HTML/CSS, без JS логики,

браузер не должен подгружать динамический JS,

браузер не должен делать запросы к backend API.

3️⃣ Серверная защита конфиденциальности

❗ Backend должен верифицировать initData от Telegram при каждом запросе.

Логика:

Если запрос не содержит валидного initData от Telegram — backend возвращает 401/403 и не отвечает данными.

Backend не отдает ни одного endpoint’а без авторизации через Telegram.

Это означает:

никакой URL API не работает → без Telegram

никакие данные не видны → без Telegram

даже ошибку 404 показывать только в безопасном формате

4️⃣ Никакой утечки через DevTools

Приложение не должно отправлять в браузер:

конфигурационные объекты со скрытыми значениями,

переменные окружения,

настройки backend,

ключи API,

любые приватные данные.

Всё, что нужно для работы UI, должно быть:

либо строго минимальный UI (если открыт в браузере),

либо отдельный, безопасный UI под Telegram Mini App.

Любые данные, которые приходят на frontend, должны быть:

привязаны к specific/signed initData,

приходить только после проверки сервера,

подписаны/шифрованы по стандартам Telegram,

без приватных значений.

5️⃣ Защита сети (CORS/RBAC)

Backend API должен:

разрешать CORS только для Telegram Mini App доменов,

запрещать любые другие cross-site запросы,

проверять каждую сессию по токену Telegram,

не отдавать ответов на прямые GET/POST запросы без Telegram авторизации.

6️⃣ Запросы/ответы должны быть безопасными

На каждом API endpoint:

Любой запрос без валидного JWT/initData → отказ.

В ответе не должно быть:

логов

данных конфигурации

разбора ошибок со стеком

нежелательных сообщений

Ошибки должны быть в нейтральном формате:

{
  "error": "Unauthorized"
}

7️⃣ Отсутствие индикаторов “тут был разработчик”

Не должно быть:

комментариев в коде с ключами

имён пользователей

usernames

чат-идов

случайных EMAIL/TELEGRAM ID

любой внутренней информации

🧪 Test Cases (как проверять)

Чтобы убедиться, что всё защищено, нужно проверить следующие кейсы:

🔹 Открытие приложения через чистый браузер

Ожидается:

только заглушка “Открой через Telegram”

ни одного сетевого вызова к API

ни одной ошибки в консоли

ни одного JS скрипта, который требует авторизацию

никакой информации о backend/логах

🔹 Открытие через DevTools “Sources”

Ожидается:

нет секретов

нет ключей

нет переменных окружения

нет backend URL

только безопасный JS/UI

🔹 Прямой запрос к API через браузер

Например:

GET https://backend/api/wallet


Ожидается:

401 Unauthorized


или

403 Forbidden

🔹 Прямой доступ к графикам/коду через браузер

Ожидается:

не выдавать никакие динамические данные

только заглушку

backend должен игнорировать

📌 Итог

Система должна быть конфиденциальной и защищённой так, чтобы:

✔ Никакие секреты не просочились в браузер
✔ В DevTools нельзя было найти ключи/токены
✔ В API нельзя попасть напрямую без Telegram initData
✔ В сети/поиске нельзя собрать информацию о твоём backend
✔ Открытие через браузер — безопасный статичный экран
